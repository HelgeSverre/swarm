Primary Recommendation

Introduce a Shared Runtime Kernel

Replace the current parent-process graph, worker-process graph, and global service singletons with one explicit composition root that builds the same service graph for both execution modes. Make runtime boundaries typed, inject policy explicitly, and move orchestration into narrow adapters.

One change, applied consistently: RuntimeKernel becomes the only place where services are wired, policies are chosen, and process/UI adapters are attached.

Why This Change First

Fixes split-brain runtime state

Parent and worker stop assembling different graphs. Both receive a RuntimeContext built from the same rules.

Centralizes file access policy

Filesystem tools no longer invent their own path rules. They all depend on one injected FileAccessPolicy.

Removes hidden globals

Eventing and logging become constructor dependencies, not static registries that vary by initialization order.

Core Interfaces

Composition Root

RuntimeKernel::forCli(AppConfig $config, ProjectContext $project)

RuntimeKernel::forWorker(AppConfig $config, ProjectContext $project, StreamSink $sink)

Both methods return the same RuntimeContext shape.

Shared Policy Contracts

FileAccessPolicy, EventPublisher, LogSink, StateStore, PromptCatalog

These replace per-class ad hoc choices and static service lookups.

Typed Boundaries

CliCommand, CommandAction, WorkerMessage, WorkerMessageType

Commands and worker updates stop being loose string protocols.

Narrower Agent Core

CodingAgent becomes a thin facade over RequestPipeline, RequestClassifier, and HandlerRegistry.

Before vs After

Concern Current Proposed
Service wiring Different graphs in Swarm, Container, and WorkerProcess One graph from RuntimeKernel
Filesystem policy Spread across tools, cwd checks, optional PathChecker One injected policy used by every filesystem tool
Events and logs Static globals with hidden lifecycle Explicit dependencies per runtime context
Command/update contracts Stringly typed and prone to drift Enums or message DTOs with one mapper
Agent coordination God module with manual handler creation Pipeline + registry with smaller services
TUI design All concerns in one class Presenter + renderer + terminal driver

Concrete Migration Sequence

  1. Build RuntimeKernel and RuntimeContext without changing behavior. Make both Swarm and WorkerProcess resolve services from it.
  2. Introduce FileAccessPolicy and inject it into ReadFile, WriteFile, Glob, and Grep. Remove duplicate cwd-based checks.
  3. Replace EventAware and Loggable globals with injected EventPublisher and LogSink. Keep adapters for temporary compatibility.
  4. Split CodingAgent into RequestPipeline, RequestClassifier, HandlerRegistry, and ErrorRecoveryPolicy.
  5. Replace string actions and update types with typed messages. Make one translation point for UI and worker transport.
  6. Break FullTerminalUI into TerminalDriver, InputController, TuiViewModel, and TuiRenderer.
